# shelter-chat-api — Architecture Walkthrough

## 1. Overall Architecture

Think of the system as three layers:

```
Browser (React)
    ↕  HTTP / SSE stream
FastAPI backend  ←→  PostgreSQL (state + data)
    ↕  MCP protocol
MCP tool server  ←→  PostgreSQL + pgvector + Ollama
```

The **browser** sends a chat message and opens a streaming connection (SSE = Server-Sent Events, like a one-way websocket). The **FastAPI backend** runs the AI pipeline and streams events back as they happen. The **MCP server** is a separate process that owns the search tools and database access.

When FastAPI starts up (`app/main.py:34-52`), it does three things in order:
1. Creates LangGraph's PostgreSQL checkpoint tables
2. Connects to the MCP server and downloads the tool list
3. Compiles the LangGraph agent with those tools

Those three objects (`checkpointer`, `mcp_tools`, `agent_graph`) are kept as module-level globals and reused for every request.

## 2. LangGraph — How It's Used

LangGraph is a **state machine**. You define a shared state dict, nodes that transform it, and edges connecting them. The compiled graph runs like a flowchart.

**The state** (`app/agent/state.py`) is a TypedDict with four fields:
- `messages` — the full chat history (uses `add_messages` so new messages are appended, not overwritten)
- `groups` — the extracted service needs
- `results` — raw search results per group
- `formatted` — final AI rationale + service IDs per group

**The graph** (`app/agent/graph.py`) defines five nodes in a fixed sequence:

```
guardrails → [conditional] → classify_groups → intake → search_per_group → format_results → END
                   ↓
                  END (if blocked)
```

The only branching is `after_guardrails`: if guardrails added an `AIMessage` to state (meaning it blocked the request), the graph ends immediately. Otherwise it continues.

**Checkpointing** is built into LangGraph. Every time a node finishes, the full state is saved to PostgreSQL under the `thread_id`. This is how conversations persist across requests — send `conversation_id` in the request and LangGraph reloads the exact state from where it left off.

**Streaming** happens via `graph.astream_events()` (`app/agent/runner.py:41-45`). This is a LangGraph API that fires events as work happens — tool calls starting/ending, nodes completing — rather than waiting for everything to finish. The runner translates those raw events into typed dicts, and the SSE generator (`app/api/chat.py:24-74`) serializes them to the browser.

## 3. How NeMo Guardrails Fits In

NeMo Guardrails is an Nvidia library that wraps an LLM with rule-based filters. It runs as the **very first node** in the graph.

The guardrails config has two files:
- `app/guardrails/config/prompts.co` — defines "flows": patterns of user messages and bot responses. Two flows: `check input safety` (harmful content) and `check input scope` (off-topic, non-SF questions)
- `app/guardrails/config/config.yml` — tells NeMo which LLM to use (Ollama, llama3.2)

The node (`app/guardrails/node.py`) works like this:
1. Pass the user's message to NeMo
2. NeMo runs it against the rules using the LLM
3. If the output **equals** the input → message passed, state unchanged, graph continues
4. If the output **differs** → that's the refusal text. It's added as an `AIMessage` to state, and the graph stops via the `after_guardrails` conditional edge

If NeMo times out (30s) or throws any error, the node **fails open** — it returns the unchanged state and the graph continues. So guardrails can never block legitimate requests due to a service outage.

## 4. pgvector — How It's Being Used

pgvector isn't directly in this repo — it lives in the MCP server (`shelter-mcp-server`). But it's what powers the actual search.

Standard SQL search is keyword-based: find rows where a column *contains* the word "shelter". That fails for natural language — "emergency housing" and "overnight stay" mean the same thing but share no words.

pgvector stores **embeddings** alongside each service record. An embedding is a list of ~768 numbers that represents the *meaning* of text (generated by Ollama's `nomic-embed-text` model). Texts with similar meaning will have numerically similar vectors.

The search flow:
1. User's query ("shelter for a single mom") → embed it → get a vector
2. SQL query using `<=>` (cosine distance operator): find services whose embedding vector is close to the query vector
3. Apply hard filters: category must match, eligibility must match, location within 50km, schedule matches
4. Sort: category match first, then eligibility, then distance

The result is semantic search: "overnight bed" matches "emergency shelter" even without shared words.

## 5. What MCP Is Doing

MCP (Model Context Protocol) is a standardized HTTP API for AI tools. Instead of each AI framework inventing its own way to call external tools, MCP defines a protocol: the client asks "what tools do you have?", gets a schema back, then calls tools by name with JSON args.

**Why a separate process?** The search logic, database connections, geocoding, and embeddings all live in `shelter-mcp-server`. Keeping them separate means the MCP server can be updated, scaled, or replaced independently of the AI pipeline.

**How it connects**: At startup, `MCPClient.connect()` calls `get_tools()` on the MCP server, which returns LangChain-compatible tool objects. These get passed into `build_graph()` and each node that needs them gets them via closure.

**The wrapping quirk** (`app/core/mcp_client.py:10-27`): MCP tool results come back as `[{"type": "text", "text": "<json string>"}]` — a list containing a content block with a JSON string inside. Before you can use the result, you have to unwrap it. That's what `_unwrap_tool_result()` does everywhere — it's just peeling that envelope open to get the actual data.

**The six tools** and where they're used:

| Tool | Used in |
|------|---------|
| `list_categories` | `intake` — to map "shelter" → `sfsg-shelter` |
| `list_eligibilities` | `intake` — to map "single mom" → `["Single Parent"]` |
| `geocode_location` | `intake` — to map "Tenderloin" → `{lat, lng}` |
| `search_services` | `search_per_group` — the actual pgvector search |
| `get_service_details` | API route `/services/batch` — frontend fetching full service cards |
| `get_service_details_batch` | Same — batch version |

## One Request, End to End

```
POST /api/v1/chat  {"message": "shelter for a single mom near Tenderloin tonight"}
  │
  ├─ auth check (Auth0 JWT or dev bypass)
  ├─ generate conversation_id (UUID)
  └─ return StreamingResponse → opens SSE stream

    graph.astream_events(...) starts:

    [guardrails_node]
      NeMo checks message → safe → state unchanged → continue

    [classify_groups_node]
      LLM + prompt → {"groups": [{group_id:1, what:"shelter", who:"single mom",
                                   where:"Tenderloin, SF", when:"tonight", open_now:false}]}
      → state.groups updated

    [intake_node]
      list_categories() → ["sfsg-shelter", "sfsg-food", ...]
      _map_categories("shelter", [...]) → ["sfsg-shelter"]

      list_eligibilities() → {family_status: ["Single Parent", ...], ...}
      _map_eligibilities("single mom", {...}) → ["Single Parent", "Anyone in Need"]

      geocode_location("Tenderloin, SF") → {lat: 37.784, lng: -122.413}

      No gaps → no interrupt → state.groups updated with categories/eligibilities/lat/lng

    [search_per_group_node]
      search_services({query:"shelter", categories:["sfsg-shelter"],
                       eligibilities:["Single Parent","Anyone in Need"],
                       lat:37.784, lng:-122.413, when:"tonight"})
      → [list of matching services]
      → state.results["1"] = [...]

    [format_results_node]
      LLM writes rationale for group 1 using top 5 service snippets
      → state.formatted["1"] = {rationale: "...", service_ids: [123, 456, ...]}
      → adds AIMessage("Here's what I found near the Tenderloin.") to messages

    runner emits: text event → format_complete event
    chat.py: saves referral to DB, saves conversation summary
    SSE: text-delta, format_complete, text-end, finish
```

The frontend gets the intro text streaming in, then receives the `format_complete` event with the `service_ids` list, and fetches full service details via `POST /api/v1/services/batch`.
